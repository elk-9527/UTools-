# 问题记录

## 架构问题：模块应直接放在 Win11 桌面上 ✅ 已修复

**现状**（旧）：当前通过 `utools.createBrowserWindow` 创建一个大的悬浮窗（820×580），5 个模块作为 `.widget-card` 用 `position: absolute` 放在这个窗口内部。

**期望**：每个模块应该是独立的、直接放在 Windows 桌面上。用户看到的是桌面上散落着 5 个小组件，而不是"打开一个大窗口，里面有 5 个卡片"。

**方案**（已实施）：
- `index.html` 里为每个模块单独创建一个 `createBrowserWindow`
- 每个独立窗口的尺寸 = 该模块的 defaultSize
- 每个窗口加载通用模板 `widget.html?module=clock`
- 模块间通信（日历点击→筛选待办、待办变化→日历重绘圆点）改用 utools.db 桥接文档轮询
- 拖拽 = 窗口 `-webkit-app-region: drag` 原生移动，JS 轮询 `screenX/Y` 保存位置
- 缩放 = JS `window.resizeTo()` + 右下角手柄
- 去掉了 `#desktop-container`、卡片绝对定位、网格吸附、`WidgetFramework` 注册表、右键菜单等概念

**注意**：`framework.js` 在新架构中已不再使用。

## 性能问题 ✅ 已修复

**现象**（旧）：拖拽卡片非常卡顿，模块伸缩也非常卡。

**原因分析**：
1. `document.mousemove` 回调在每次像素移动时都触发（60+ Hz），其中包含 `_snap()` 遍历所有模块实例
2. 卡片 CSS 有 `transition` 属性，拖拽时浏览器在计算动画
3. `backdrop-filter: blur()` 在每次重绘时被重新计算，GPU 开销大

**修复方案**（已实施）：
1. mousemove 用 `requestAnimationFrame` 节流
2. 拖拽期间 `.dragging` 类禁用 transition
3. 缓存其他卡片位置避免 parseInt
4. 架构改为独立窗口后，拖拽由 OS 原生窗口管理器处理，不再需要 JS 监听 mousemove 更新 DOM 位置

## 架构改动影响

以下文件在重构中发生**重大变更**：
- `index.html` — 从创建 1 个窗口改为创建 5 个独立窗口
- `widget.html` — 从大容器（含所有模块的注册/右键菜单/设置面板）改为单模块通用模板
- `style.css` — 删掉了所有 `.widget-card` / `.widget-header` / `.resize-handle` / `#desktop-container` / `#add-module-btn` 等卡片布局样式，只保留主题变量、body 毛玻璃、通用组件（按钮/输入框/滚动条/右键菜单）
- `framework.js` — 新架构不再使用（各窗口独立管理生命周期）
- `calendar.js` — CustomEvent 改为 db 桥接轮询
- `todo.js` — CustomEvent 改为 db 桥接轮询
- `preload.js` — 改为用 uTools 内置 API（`utools.shellOpenExternal`、`utools.shellOpenPath`）代替 Node.js `exec`

未变更：`clock.js`、`weather.js`、`quick-launch.js`、各模块 `.css` 文件、`plugin.json`
